What is open source?

Before we dive into the main content of the presentation. Let us have a quick review of "What is open source".

Access

And the first keyword defines open source is who can access the code. If the author put the source code of the project on the internet for everyone to read, it is usually considered open source. The usages are often protected by their open-source licenses. For example, some of the licenses may prevent you from using their code to make money.

Umidigi F2 kernel source code

However, there are always corner cases. The Umidigi F2 is a fork of the Linux kernel. And the Linux kernel is protected by the GPLv2 open-source license that requires all the code using it to be open-source. So, they have to open source their code according to the license. One day a user send an email to request their code since they could not find it on the Internet. Guess what reply the got?

"You can request the shareable source code at our office."

Ben, UMIDIGI

This is the reply they got. "You can request the shareable source code at our office." So they may need to apply their visa and book a air ticket and hotel to get the source code. And with all these intentionally created difficulties, the code is still considered open source. Pretty interesting.

Why do we open source?

Why do we open source?
Let's use Android again as an example to illustrate this.
When we use Android, we usually use it combined with the Google Play Store, and many other services provided by Google. Android is open source. So the smartphone companies want to use it. It attracted a large number of users of this open-source operating system. Many of them would use the operating system combined with the Google services like the playstore. And for every dollar they spent in the Play Store, Google gets a share.

Why open source TensorFlow?

So why was TensorFlow open sourced in the first place? I do not have the exact information but I believe it was a mix of factors.

  • Professional branding

First, it is about professional branding. Let everyone working in machine learning use technologies from Google. It will project Google as a leader of the machine learning revolution. It helps Google recruit the best talents on the job market.

  • Professional branding
  • Keep engineers happy

Secondly, open source keeps the engineers happy. It gains exposure to the work of the engineers. It lowers the attrition rate and also helps recruiting as well.

  • Professional branding
  • Keep engineers happy
  • Full control over its infra

The third one, which I believe is the most important one, is about having full control over its machine learning infrastructure. If TensorFlow is not open-source, there will be some other framework out there gaining popularity in the community. People at Google may want to use that. Then Google lose full control over its infra. We will talk more about this in a bit.

  • Making money
  • Professional branding
  • Keep engineers happy
  • Full control over its infra

TensorFlow's blunders

Let's also look at the other side. What did TensorFlow do wrong? The general user experience for TensorFlow 2 is OK. But there are a few basic things that TensorFlow did wrong that we want to learn from.

TensorFlow's blunders

Installation

The first one is the installation guide. This is the very first step of a new user do with your open source software. It is really important but often overlooked. The installation guide for GPU on the official TensorFlow website had been broken for years. Then, they hired me by chance, when I was a new grad. I coordinated multiple teams to get it resolved. Now, you can easily install it with a one-liner.

TensorFlow's blunders

Backword compatibility

The second one is the backward compatibility. TensorFlow 2 is not compatible with TensorFlow 1. The APIs changed completely. The users needed to spend a lot of money to upgrade it. It causes a lot of discontent among the users. They would move to any framework other than TensorFlow. So installation and backward compatibility, these are important things you need to get right to have a good user experience before you do any fancy stuff.

The 3 simple principles

Now, I will present three simple yet powerful principles to make UX better. These are the most valuable knowledge I learned through the past few years working on Keras.

Before we dive in a few quick updates from the Keras team. Keras 3 has just been released with game-changing new features. Keras has been around for a comparatively long time as a developer interface sits on top of the deep learning frameworks. It supported Theano, MXNet, and TensorFlow.

Keras 3

  • Support TensorFlow, JAX, PyTorch

Now, with Keras3, we officially support 3 backends: TensorFlow, JAX and PyTorch. Now, with one set of code, you get the benefits from all frameworks. The eager developing and debugging experience of PyTorch. Tap into the comprehensive tooling of the TensorFlow ecosystem. Faster training with JAX on TPU. Because of these benefits, it got adopted by MidJourney right away. They want the deployment path with PyTorch and want faster training with JAX.

Keras 3

  • Support TensorFlow, JAX, PyTorch
  • Backward-compatible & future-proof

It is backward-compatible and future-proof. The Keras API has not been changed much ever since its very first release. You can easily migrate your code to Keras 3 with no changes unless you did something really special. Keras will continue to support any new backends in the future. If one day PyTorch is gone, and a new framework comes along, you still don't need to change your code.

The 3 simple principles

from Keras

Keras is a high-level API, and it is the best of its kind if considered independent of the underlying frameworks. And the high-level API is just about user experience. Here are the 3 simple principles that we followed to ensure this user experience.

Principle 1:

Design end-to-end workflows

Principle number one, design end-to-end workflows. So what does it mean? 

				class Model:
					def __call__(self, input):
						"""The forward call of the model.

						Args:
							input: A tensor. The input to the model.
						"""
						pass
When we think of designing the APIs of a piece of software, you may look like this. Define the class and add the documentations. However, this would not help us understand much about the user experience. 

				input = keras.Input(shape=(10,))
				x = layers.Dense(32, activation='relu')(input)
				output = layers.Dense(10, activation='softmax')(x)
				model = keras.models.Model(inputs=input, outputs=output)
				model.compile(
					optimizer='adam', loss='categorical_crossentropy'
				)
What we should do is something like this. We want to write out the entire user workflow of using the software. Idealy, it should be a tutorial of how to use the software. It provides much more information about the user experience. 

				class RandomSearch:
					def __init__(self, ..., metrics, objective="val_loss", ...):
						"""The initializer.

						Args:
							metrics: A list of Keras metrics.
							objective: String or a custom metric function. The
								name of the metirc we want to minimize.
						"""
						pass
Let's look at another example. This is from KerasTuner. We use this RandomSearch class to select the best model, which is the one with the smallest validation loss. Again, it doesn't provide much info about the user experience. 

				tuner = RandomSearch(
					...,
					metrics=[custom_metric],
					objective="val_???",
				)
However, if we write an end-to-end workflow like this. The user try to define a custom metric. The objective is not so straight-forward to use any more. What's its name now? We can easily spot a user experience problem by writing this workflow. 

				tuner = RandomSearch(
					...,
					metrics=[custom_metric],
					objective="val_custom_metric",
				)
In fact, it should be just "val_custom_metric". Obviously, it is not intuitive enough. We want to make it better instead of forcing the user to learn this. 

				def custom_metric(y_true, y_pred):
					squared_diff = ops.square(y_true - y_pred)
					return ops.mean(squared_diff, axis=-1)

				tuner = RandomSearch(
					...,
					metrics=[custom_metric],
					objective="val_custom_metric",
				)
Another thing is that, if you wrote the design more comprehensively, you will find you even need to learn how to write a keras custom metric. 

				class MyHyperModel(HyperModel):
					def fit(self, trial, model, validation_data):
						x_val, y_true = validation_data
						y_pred = model(x_val)
						return custom_metric(y_true, y_pred)

				tuner = RandomSearch(MyHyperModel(), max_trials=20)
So we specially designed a better workflow for custom metrics. You only need to override the `fit` function, compute your custom metric and return it. The UX is much better right now.

UX Implementation

One more thing to remember is we always start from the UX. The designed workflows deside the implementation.

Principle 2:

Minimize cognitive load

Principle number two, minimize the cognitive load for the user when learning to use your software. Do not force the user to learn anything unless it is really necessary. Let's see some good examples.

Good example: Keras modeling


				model = keras.Sequential([
					layers.Dense(10, activation="relu"),
					layers.Dense(num_classes, activation="softmax"),
				])
				model.compile(loss='categorical_crossentropy')
				model.fit(...)
				model.predict(...)
One good example is the Keras modeling API, which is pretty intuitive. The model builders already have these concepts in mind, like a model is a stack of layers. It needs a loss function. We can fit it with data or make it predict on data. So basically, no new concepts learned in order to use Keras.

Good example: PyTorch modeling


					class MyModel(nn.Module):
						def forward(self, x):
							if x.sum() > 0:
								return self.path_a(x)
							return self.path_b(x)
Another good example is the PyTorch modeling. The code is executed just like Python code. All tensors are just real tensors with real values. You can depend on the value of a tensor to decide your path with plain Python code.

Bad example: Keras with JAX or TensorFlow


					class MyModel(keras.Model):
						def call(self, x):
							return ops.cond(
								ops.sum(inputs) > 0,
								lambda : self.path_a(inputs),
								lambda : self.path_b(inputs),
							)
You can also do this with Keras with TensorFlow or JAX backend, but needs to be written in a different way. All the if conditions need to be written with this ops.cond function. This is less intuitive which is bad. In compensation, it brings significant improvement in training speed.

Bad example: PyTorch memory and speed optimization


				torch.relu(x, inplace=True)
				x = torch._foreach_add(x, y)
				torch._foreach_add_(x, y)

Bad example: PyTorch compute gradients


				inputs, labels = batch
				outputs = model(inputs)
				loss = torch.nn.functional.cross_entropy(outputs, labels)
				optimizer.zero_grad()
				loss.backward()
				optimizer.step()
loss.backward computes the gradients. This is introducing a new concept of actually computing the gradients. I don't think the user has to know about it.

Good example: keras.ops, tensorflow.numpy, jax.numpy

When you have to introduce some cognitive load, just reuse what people already familiar with.

Worst case: trick the users

The worst thing you can do with user experience is to trick the users. Trick the user to believe your API is something else than what it really is.

Really bad example: How to pad the images?

				# pad the 32x32 images to (1+32+1)x(2+32+2)
				# (100, 3, 32, 32) to (100, 3, 34, 36)

Really bad example: How to pad the images?

				# pad the 32x32 images to (1+32+1)x(2+32+2)
				# (100, 3, 32, 32) to (100, 3, 34, 36)
				out = F.pad(
					torch.empty(100, 3, 32, 32),
					pad=((0, 0), (0, 0), (1, 1), (2, 2)),

Really bad example: How to pad the images?


				import torch.nn.functional as F
				# pad the 32x32 images to (1+32+1)x(2+32+2)
				# (100, 3, 32, 32) to (100, 3, 34, 36)
				out = F.pad(
					torch.empty(100, 3, 32, 32),
					pad=(2, 2, 1, 1),

The correct answer is this. You need to reverse the order of the sequence.
